## Summary

This branch operationalises four of the design decisions recorded in `planning/PLAN.md` Section 13 and addresses prior code-review findings on the market data subsystem:

1. SSE contract trimmed to the agreed five fields, with daily-change `%` derived client-side from a new `open_price` reference (PLAN §10 / §13 "Daily change % source").
2. Push-on-change SSE with per-ticker version diffing and a 10s comment-only keepalive (PLAN §13 "SSE cadence vs. Massive API polling interval").
3. Rolling 200-point price history per ticker exposed via `PriceCache.get_history()` — the data layer for the planned `GET /api/prices/{ticker}/history` endpoint (PLAN §13 "Missing ticker history endpoint").
4. Cache hardening: timestamp falsy-zero bug fixed, `remove()` now wipes all per-ticker state (prices, opens, history, version) and bumps the global version.

Plus tactical fixes — case normalisation in `SimulatorDataSource.add_ticker` / `remove_ticker`, a `LinAlgError` fallback in the Cholesky path, conversion of `stream.py` from a module-level router to a factory, and a new `test_stream.py` with 9 tests (including the SSE-pruning mid-stream path).

Overall this is a clean, well-scoped commit that lands several decisions consistently across model, cache, stream, tests, and developer docs. Test discipline is high — every behavioural change has at least one targeted assertion, including the subtle ones (timestamp zero, per-ticker version isolation, oldest-first history ordering, pruning of removed tickers from the SSE diff state).

### M1 — SSE generator opens a small race between `get_ticker_versions()` and `get(ticker)`

`backend/app/market/stream.py:82-90`:

```python
current_versions = price_cache.get_ticker_versions()

changed: dict[str, dict] = {}
for ticker, version in current_versions.items():
    if version != last_seen_versions.get(ticker, -1):
        update = price_cache.get(ticker)
        if update is not None:
            changed[ticker] = update.to_sse_dict()
        last_seen_versions[ticker] = version
```

`get_ticker_versions()` and `get(ticker)` each take the cache lock independently. Between the two calls, a writer can `remove(ticker)`. In that case:

- The ticker is present in `current_versions` (captured a moment ago)
- `get(ticker)` returns `None`
- The branch falls through without yielding, **but** `last_seen_versions[ticker] = version` is still recorded
- Then the pruning loop runs and deletes it from `last_seen_versions` anyway (because the ticker is also gone from `current_versions`… wait — `current_versions` was the *captured* snapshot, so it's still in there)

Trace it: after the loop, `current_versions` still contains the removed ticker, so the prune step (`set(last_seen_versions) - set(current_versions)`) does **not** drop it. On the very next iteration `get_ticker_versions()` no longer contains the ticker, so the prune step will catch it then. No correctness bug — but the version is briefly stale by one tick, and a removed-then-re-added ticker within a single iteration window could be skipped.

This is a low-probability edge case for a single-writer/single-reader-per-client design. Two cheap improvements:

1. Snapshot versions and updates under a single lock acquisition by adding a `PriceCache.get_changes_since(last_seen: dict[str, int]) -> dict[str, PriceUpdate]` method.
2. Or at minimum, skip writing `last_seen_versions[ticker]` when `update is None`, so the next iteration definitely re-checks.

The current behaviour is acceptable for a demo, but option 2 is a one-line tightening worth doing.

### M2 — Per-ticker version counter is never reset on `remove()` + re-add

`backend/app/market/cache.py:100-107`: `remove()` deletes `self._ticker_versions[ticker]`. A subsequent `update(ticker, …)` re-initialises the counter at `1` via `self._ticker_versions.get(ticker, 0) + 1`. That's correct.

However, an SSE client that held `last_seen_versions[ticker] = 47` before the removal will, after the prune step in `stream.py:93-94`, drop the entry — which is correct. Verified by the new `test_removed_ticker_pruned_from_stream`. No bug, but worth noting that the design relies on the prune step running before the new version (`1`) is compared against the old one (`47`); if the prune step were ever removed, this would silently mis-emit (`1 != 47` would always be true on every iteration). Consider a comment in `stream.py` documenting the prune-then-diff invariant.

### M3 — `open_price` is never re-seeded; long sessions drift away from any "daily" anchor

By design (PLAN §13 "Daily change % source"), `open_price` is the first price the cache sees for a ticker and is frozen for the life of the process. That is fine for a short demo session, but:

- A user who leaves the browser open across the simulated "next day" will see daily-change `%` that no longer corresponds to the current day's open.
- There is no API or timer to roll `open_price` over.

PLAN explicitly accepts this for now, so this is informational, not a blocker. If a "daily roll" is ever added, the natural touchpoint is `PriceCache` and it should bump per-ticker versions so all SSE clients receive the new `open_price`. Worth a follow-up TODO comment near `_open_prices` in `cache.py`.

### L4 — `GBMSimulator.add_ticker` does not normalise case itself

`SimulatorDataSource.add_ticker` / `remove_ticker` correctly normalise to upper-stripped strings before delegating to `GBMSimulator`. But the inner `GBMSimulator.add_ticker` and `_add_ticker_internal` accept whatever is passed. If anything in the future calls `GBMSimulator.add_ticker("aapl")` directly (e.g. a test, a debug tool, the planned terminal demo), the simulator will hold the lowercase variant while the cache and watchlist hold uppercase, and the watchlist→simulator wiring becomes order-sensitive.

The cheap belt-and-braces fix is to add `ticker = ticker.upper().strip()` at the top of `GBMSimulator.add_ticker` and `GBMSimulator.remove_ticker` as well. The wrapper's normalisation already handles the production path, so this is nice-to-have.

### L5 — `_KEEPALIVE_INTERVAL` is module-level and patched by a test

`backend/tests/market/test_stream.py:133-145` patches `stream_module._KEEPALIVE_INTERVAL = 0.1` to make the keepalive test fast, then restores it. This works but is the only test that mutates module state. Two cleaner options:

1. Use `monkeypatch.setattr(stream_module, "_KEEPALIVE_INTERVAL", 0.1)` so pytest restores it on any exit path (including failures inside the `try`).
2. Promote `_KEEPALIVE_INTERVAL` and `_POLL_INTERVAL` to parameters of `_generate_events()` with module defaults.

Option 1 is a two-line change.

### L6 — Drop redundant `last_emit_at = now` reset

`backend/app/market/stream.py:97-101`:

if changed:

## Strengths Worth Calling Out

- **The new `to_sse_dict()` / `to_dict()` split** is the right shape: SSE is a network contract and stays minimal; `to_dict()` is for richer JSON responses where computed properties are useful. This is the kind of disciplined boundary that pays off later.
- **Test for `test_timestamp_zero_not_replaced`** (`test_cache.py`) directly pins the bug the changelog claims to fix (`ts = timestamp if timestamp is not None else time.time()` — the old `or` would have replaced `0.0` with `time.time()`). Good regression test.
- **`test_removed_ticker_pruned_from_stream`** is exactly the right test for the SSE diff loop and the prune step. Tricky behaviour, well covered.
- **CLAUDE.md "Wiring requirement" block-quote** is the right level of forcefulness for a contract that has no automated enforcement. Future watchlist-handler implementers will see it.
- **Cholesky `LinAlgError` fallback** is a small but real improvement — adding an arbitrary new ticker to a non-default group could in principle yield a non-PSD matrix; falling back to uncorrelated moves is the right "keep running, log a warning" choice.
- **97% coverage on the touched code** with the misses being `logger.exception` and `raise` paths is appropriate.
