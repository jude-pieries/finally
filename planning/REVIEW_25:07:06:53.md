## 2. Architecture Assessment

### What is genuinely well done

1. **The `open_price` semantics are correctly enforced.** `cache.py:40-42` sets the open price exactly once per ticker on the first `update()` call and never changes it. Every subsequent `PriceUpdate` carries the frozen open price, and `to_sse_dict()` (`models.py:47-59`) includes it in every SSE payload. This implements PLAN.md section 13's decided contract precisely.

2. **Per-ticker version counters are the right SSE primitive.** `cache.py:25, 61` increments `_ticker_versions[ticker]` on every update; `stream.py:82-90` keeps a `last_seen_versions: dict[str, int]` per connection and emits only tickers whose counter advanced. This is the push-on-change behaviour specified in PLAN.md section 6 ("Push-on-change only … flash animations only fire on real price changes"), and it is O(n_tickers) per poll. No stale-price spam.

3. **Lock discipline is correct and consistent.** Every read and write in `PriceCache` is guarded by `self._lock` (`cache.py:36, 67, 72, 86, 97, 102, 112, 116, 120`). Crucially, the previously unsynchronised `version` property is now under the lock (`cache.py:112`) — the diff explicitly calls this out as "forward-compatible with free-threaded CPython". This matters once the GIL becomes optional.

4. **Router factory pattern, no module-level singleton.** `stream.py:22-54` creates a fresh `APIRouter` per call. The diff removes the module-level `router = APIRouter(...)` that existed on `main`. This avoids the duplicate-route registration bug under FastAPI hot reload and is friendlier to per-test isolation.

5. **SSE generator pruning of removed tickers.** `stream.py:92-94` correctly removes versions for tickers that have disappeared from the cache (e.g., after `cache.remove("AAPL")`). The set difference is small and runs every poll — cheap.

6. **`to_sse_dict()` enforces the 5-field contract explicitly.** Rather than relying on the larger `to_dict()`, `models.py:47-59` returns exactly `{ticker, price, previous_price, open_price, timestamp}`. The test `test_sse_event_has_exactly_five_fields` (`test_stream.py:73-81`) and `test_to_sse_dict_has_exactly_five_fields` (`test_models.py:74-77`) lock this in. A future field addition to `to_dict()` cannot accidentally leak into the SSE wire format.

7. **Immutability and slots.** `@dataclass(frozen=True, slots=True)` on `PriceUpdate` means cache reads return values that are safe to hand to multiple readers without defensive copying. `test_immutability` verifies the freeze.

8. **History buffer uses `deque(maxlen=200)`.** `cache.py:57` constructs the deque with the cap once; subsequent appends auto-evict. No periodic pruning code, no off-by-one risk. `test_history_capped_at_200` verifies that 250 appends produce exactly 200 entries.

9. **Disconnect handling.** `stream.py:77-79` polls `request.is_disconnected()` each loop iteration; `test_stops_on_disconnect` verifies that a disconnected request yields only the retry directive and exits. `asyncio.CancelledError` is caught and logged in the generator's `except` block (`stream.py:105-106`).

### What is structurally sound but should be noted

- **Two coupled 0.5-second constants.** `stream.py:19` `_POLL_INTERVAL = 0.5` and `simulator.py` `update_interval = 0.5` are independently defined but must remain approximately equal (slower SSE would starve updates; faster SSE would spin uselessly). This coupling is implicit. See Issue #6.

- **Pre-`start()` behaviour differs between sources.** `MassiveDataSource.add_ticker` appends unconditionally; `SimulatorDataSource.add_ticker` silently no-ops if `self._sim is None`. See Issue #5.

## 3. Issues Found

### Issue 1 — Bug: falsy-zero timestamp in `PriceCache.update`
**Files:** `backend/app/market/cache.py:37`
**Severity:** Low (latent — never triggered today, but reachable from the Massive client)

```python
ts = timestamp or time.time()
```

If a caller passes `timestamp=0.0`, the `or` evaluates `0.0` as falsy and falls through to `time.time()`. The Massive client computes `snap.last_trade.timestamp / 1000.0` — a malformed/empty trade payload could yield exactly `0.0`, which would silently be replaced by wall-clock time. The bug also bites if a test ever wants to exercise `timestamp=0.0`.

**Recommended fix:**
```python
ts = timestamp if timestamp is not None else time.time()
```
Add a regression test: `cache.update("X", 100.0, timestamp=0.0)` must produce a `PriceUpdate` with `timestamp == 0.0`.

The in-branch `MARKET_DATA_REVIEW.md` (Issue #1) flagged this bug. The code on this branch was not changed to fix it; both reviewer commits only added the document.

### Issue 2 — Design contract drift: explicit `add_ticker` / `remove_ticker` vs. PLAN.md "re-read on each poll"
**Files:** `backend/app/market/simulator.py:242-255`, `backend/app/market/massive_client.py:66-76`, `backend/app/market/interface.py:42-53`
**Severity:** Architectural — must be communicated to downstream agents before the watchlist REST handlers are built.

PLAN.md section 13 (decided question, Section 6 — Dynamic tickers in the price cache):

> "The background task re-reads the watchlist from the database on each poll cycle. No explicit signaling needed."

The implementation does not do that. Both sources hold an in-memory ticker set and require explicit `await source.add_ticker(t)` / `await source.remove_ticker(t)` calls. There is no SQLite read in either background loop and the `app/market/` package has no database dependency at all.

The implementation choice is **defensible** — coupling a 500 ms tick to a synchronous SQLite read on every poll would be wasteful, and the abstract interface (`interface.py:42-53`) is clean and well documented. But the divergence from the written plan is real, and the next agent who reads PLAN.md and writes `POST /api/watchlist` will issue `INSERT INTO watchlist (...)` and expect prices to flow. They won't.

**Recommended fix:** Either (a) update PLAN.md section 13 to reflect the explicit-signalling choice, or (b) add a paragraph to `backend/CLAUDE.md` stating: "Watchlist REST handlers must call `market_data_source.add_ticker(t)` and `market_data_source.remove_ticker(t)` in the same request that modifies the database. There is no background reconciliation loop." `backend/CLAUDE.md` already describes the `add_ticker / remove_ticker / start / stop` lifecycle — adding one sentence about the wiring contract closes the gap.

### Issue 4 — Ticker normalisation inconsistency between implementations
**Files:** `backend/app/market/massive_client.py:66-74`, `backend/app/market/simulator.py:242-255`
**Severity:** Low — but a real behavioural difference.

```python
# MassiveDataSource
async def add_ticker(self, ticker: str) -> None:
    ticker = ticker.upper().strip()         # normalised

# SimulatorDataSource
async def add_ticker(self, ticker: str) -> None:
    if self._sim:
        self._sim.add_ticker(ticker)        # NOT normalised
```

A caller passing `"  aapl  "` ends up with `AAPL` in the Massive cache but `  aapl  ` (a distinct key) in the simulator. The simulator's `test_add_ticker_uppercase_normalization` and `test_add_ticker_strips_whitespace` tests in `test_massive.py` confirm the Massive side strips/uppercases; there is no equivalent enforcement on the simulator side. The same asymmetry exists for `remove_ticker`.

**Recommended fix:** Either (a) normalise at the watchlist API boundary and document the contract as "the data source receives canonical uppercase tickers", or (b) push normalisation into the abstract `MarketDataSource` base class so both implementations behave identically. (a) is cheaper; (b) is more defensive.

### Issue 5 — Pre-`start()` behaviour differs between sources
**Files:** `backend/app/market/simulator.py:242`, `backend/app/market/massive_client.py:66`
**Severity:** Very low — undefined-behaviour territory regardless, but worth aligning.

```python
# Simulator: silent no-op before start()
async def add_ticker(self, ticker: str) -> None:
    if self._sim:
        ...

# Massive: appends to the in-memory list before start()
async def add_ticker(self, ticker: str) -> None:
    ticker = ticker.upper().strip()
    if ticker not in self._tickers:
        self._tickers.append(ticker)
```

The Massive variant buffers; the simulator variant silently discards. The two implementations advertise the same `MarketDataSource` contract but observably differ here.

**Recommended fix:** Either (a) make both raise `RuntimeError("add_ticker called before start()")`, or (b) make both buffer until `start()`. Avoid silent asymmetric failure.

### Issue 6 — SSE poll interval and simulator update interval are implicitly coupled
**Files:** `backend/app/market/stream.py:19` (`_POLL_INTERVAL = 0.5`), `backend/app/market/simulator.py` (default `update_interval = 0.5`)
**Severity:** Informational — no bug today.

Both constants are `0.5` and were independently chosen. They must remain approximately aligned: an SSE poll significantly slower than the source tick under-samples updates; significantly faster wastes CPU. Today, in the worst case, a simulator tick at time `t` is visible to an SSE client at `t + 0.5 s` (~1 s of visible latency in the demo) — fine for the spec.

**Recommended fix:** A one-line comment on `stream.py:19` noting the relationship to `SimulatorDataSource.update_interval`, or a shared `MARKET_TICK_INTERVAL` constant exported from `app/market/__init__.py`.

### Issue 8 — Minor: keepalive test mutates a module-level constant
**File:** `backend/tests/market/test_stream.py:144-158`

```python
from app.market import stream as stream_module
original = stream_module._KEEPALIVE_INTERVAL
try:
    stream_module._KEEPALIVE_INTERVAL = 0.1
    ...
finally:
    stream_module._KEEPALIVE_INTERVAL = original
```

This works because Python rebinds the module attribute and the generator reads `_KEEPALIVE_INTERVAL` by name each iteration via the module namespace... except it doesn't — `stream.py:99` references the bare name `_KEEPALIVE_INTERVAL`, which under Python's `LOAD_GLOBAL` does in fact pick up the rebound module attribute. The test passes, but the pattern is fragile (any future refactor that captures the constant into a local at function entry would silently break the test). Consider using `monkeypatch.setattr(stream_module, "_KEEPALIVE_INTERVAL", 0.1)` instead — pytest's `monkeypatch` fixture handles teardown automatically and is the conventional idiom.

**Severity:** Style — the test is correct today.
